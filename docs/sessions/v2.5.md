## 1. Trailing Stop Fix

**Problem**: Trailing stop would get "stuck" at break-even price. When price dropped to trigger the trailing stop, the condition `current_price >= trailing_stop` (at break-even) would fail because break-even was being used as the floor, creating a circular logic trap.

**Root cause** (`src/trading_engine.py`, `monitor_positions()`):
- Trailing stop was calculated as `peak × (1 - trailing_pct)`, floored at `break_even`
- When price dropped to break-even, `current_price >= break_even` was True
- But `peak_price` only updated when `current_price > peak_price`
- When price dropped to trailing (which equaled break-even), sell check would fail
- Position wouldn't close even though it should have

**Fix**:
```
Trailing floor at entry price (not break-even)
Sell triggers when: current_price >= entry_price (covers fees via break_even check first)
```

| Condition | Old | New |
|-----------|-----|-----|
| Trailing floor | `max(trailing_stop, break_even, initial_stop)` | `max(entry_price, ...)` |
| Sell trigger | `current_price < final_stop` | `current_price < trailing_stop` |
| Stop check | `trailing_stop = max(trailing_stop, break_even)` | Trailing stops only when above entry |

**Code change** (`src/trading_engine.py`, ~line 540):
```
OLD: if current_price < final_stop:
      if current_price >= break_even:
          # Never sell at loss - wait for recovery
NEW: if current_price < trailing_stop:
      # Sell to lock in profit, even at entry price
      if current_price >= entry_price:
          sell = True
```

## 2. LR Model Removal

**Problem**: LR (Logistic Regression) was severely underperforming — average confidence ~23%, consistently worse than RF/MLP/GB. It was dragging down ensemble confidence and adding noise to agreement calculations.

**Fix** (`config/settings.py`):
```python
USE_LR_MODEL = False  # Was True
```

**Code impact** (`src/ai_model.py`):
- `USE_LR_MODEL = False` at inference → LR model loaded but excluded from ensemble voting/weights
- LR still trained during `find_best_atr_config()` (walk-forward trains all 4 models regardless)
- Kept in training pipeline in case of future re-enable

## 3. Confidence Threshold Fix

**Problem**: 80% confidence threshold was too strict. Most signals had high agreement (67-100%) but individual model confidence scores were just under 80%, blocking nearly all signals.

**Fix** (`src/ai_model.py`, `get_signal()`):

Added "unanimous agreement" exception:
```
if conf > 80% threshold OR (unanimous AND conf >= 50%):
    action = BUY/SELL
else:
    action = HOLD
```

| Condition | Result |
|-----------|--------|
| Confidence ≥ 80% threshold | Execute signal |
| All models agree (unanimous) + confidence ≥ 50% | Execute signal (exception) |
| Otherwise | HOLD |

**Agreement calculation** (`src/ai_model.py`, line 884):
- Weighted by distance from majority vote
- HOLD is "close" to both SELL and BUY (0.75 weight)
- Full agreement = 1.0, partial agreement weighted proportionally

## 5. SyntaxError Fix

**Problem**: Container failed to boot — gunicorn workers crashing with `SyntaxError: expected 'except' or 'finally' block` at api_worker.py:876.

**Root cause**: The nested `try/except` for accuracy loading closed the outer `try` prematurely:
```
for product_id in PRODUCT_IDS:
    try:                   # Line 803 - outer try
        ...
        try:               # Line 815 - nested try
            ...
        except Exception as e:  # Line 834 - CLOSES outer try!
            logger.warning(...)
        ...
    except Exception as e:  # Line 876 - ORPHANED! No matching try
```

**Fix** (`src/api_worker.py`, line 876):
```python
OLD: except Exception as e:
NEW:             except Exception as e:  # Indented to match try at line 803
```
